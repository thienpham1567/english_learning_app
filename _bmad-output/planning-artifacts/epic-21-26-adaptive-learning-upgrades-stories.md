# english_learning_app - Adaptive Learning Upgrade Epic Breakdown

## Overview

This document decomposes the next product upgrade for the English self-learning web app into BMAD-style epics and stories. The current app already has broad learning coverage across vocabulary, grammar, listening, reading, writing, speaking, pronunciation, mock tests, daily challenges, error notebook, analytics, and AI tutoring. Epic 20 has also added the personalization foundation: learning events, skill taxonomy, mastery state, recommendation scoring, daily study plan generation, review task models, AI feedback metadata, and onboarding baseline.

The next upgrade should not add more disconnected learning modules. It should connect the existing modules into an adaptive learning system that tells each learner what to study today, explains why, routes them to the right activity, captures the outcome, and updates future recommendations.

## Requirements Inventory

### Functional Requirements

FR1: Home SHALL consume the adaptive daily study plan instead of building the primary plan manually from dashboard data.

FR2: Daily plan items SHALL include title, reason, estimated minutes, skill ids, priority, completion state, and an actionable route.

FR3: Recommendation action URLs SHALL be audited against real app routes, with automated coverage for invalid internal links.

FR4: The app SHALL provide one "Today's Review" experience that can combine vocabulary, flashcards, grammar mistakes, writing errors, pronunciation issues, listening misses, and reading cloze misses.

FR5: Review completion SHALL update the unified review task, create a learning event, and refresh mastery state when possible.

FR6: Error Notebook SHALL evolve from a list of mistakes into a personal remediation center with error categories, retry drills, and evidence of improvement.

FR7: Writing, speaking, grammar, listening, reading, and mock-test feedback SHALL be normalized into reusable structured findings where feasible.

FR8: New or returning learners SHALL be guided through goal, time budget, weak skill, and optional diagnostic setup.

FR9: The onboarding or diagnostic result SHALL seed baseline skill state and generate a short starter pathway.

FR10: The app SHALL provide a 7-day adaptive starter pathway for major learner goals: TOEIC, IELTS, daily conversation, workplace English, and foundation rebuilding.

FR11: End-of-lesson screens SHALL follow a consistent result, explanation, and next-action pattern.

FR12: The chatbot SHALL act as a learning coach by summarizing recent progress, suggesting next actions, and linking to relevant modules.

FR13: AI feedback outputs used for learning decisions SHALL be schema-validated, versioned, and safe to convert into review tasks.

FR14: Progress reporting SHALL explain mastery, weak spots, repeated errors, review debt, and next actions using learner-facing language.

FR15: Motivation features SHALL emphasize visible English improvement, not only XP, streaks, badges, or leaderboard position.

FR16: Existing module behavior SHALL remain backward compatible while upgrades are introduced incrementally.

### NonFunctional Requirements

NFR1: API routes should stay thin; learning logic should remain in packages where practical.

NFR2: New data model changes must be additive and must not rewrite historical activity data.

NFR3: Daily plan and dashboard requests should remain fast enough for the Home page to feel immediate.

NFR4: Recommendation decisions must be deterministic and explainable before introducing model-based ranking.

NFR5: Learning event and review task writes should not noticeably slow critical user flows.

NFR6: New UI should preserve the current Ant Design and shared component conventions.

NFR7: All high-risk recommendation, review, mastery, and feedback rules require unit or route tests.

NFR8: Internal navigation links generated by recommendation services must be covered by tests.

### Additional Requirements

- Preserve the existing Next.js app and package architecture.
- Build on existing `learning_event`, `user_skill_state`, `review_task`, `ai_feedback_run`, and `onboarding_baseline` tables.
- Preserve existing vocabulary SRS, flashcard review, error notebook, and dashboard behavior while migrating to unified flows.
- Fix known recommendation route mismatch where speaking currently maps to `/chatbot` instead of the actual app route.
- Avoid adding a new learning module until the existing modules are connected into a coherent adaptive loop.

### UX Design Requirements

UX-DR1: Home should answer "What should I study today?", "Why this?", and "How long will it take?" before secondary widgets.

UX-DR2: Review should be presented as learner needs, not as disconnected module names.

UX-DR3: Onboarding should reduce choice overload by showing a focused pathway after goal selection.

UX-DR4: End-of-lesson feedback should use the same structure across modules: result, explanation, next action.

UX-DR5: Progress should show meaningful improvement statements such as remembered words, reduced repeated errors, higher reading level, longer speaking duration, or better listening speed.

UX-DR6: Advanced features should remain discoverable but should not compete with the primary daily study action for new learners.

## Epic 21: Adaptive Home and Daily Session

Learners open the app and immediately know what to study today, why it matters, and where to begin.

### Story 21.1: Audit Recommendation Routes

As a learner,
I want recommendation links to open the correct app module,
So that I can trust the daily plan and continue learning without dead ends.

**Acceptance Criteria:**

**Given** the app has generated recommendation candidates
**When** a candidate includes an internal `actionUrl`
**Then** the URL matches an existing app route
**And** the speaking recommendation no longer points to `/chatbot`
**And** a route coverage test fails if a recommendation URL does not map to a known route
**And** existing recommendations for vocabulary, grammar, listening, reading, writing, pronunciation, and exam strategy still route correctly.

### Story 21.2: Add Daily Plan Client Hook

As a learner,
I want Home to load my adaptive daily study plan,
So that the plan is based on my actual mastery state and goals.

**Acceptance Criteria:**

**Given** an authenticated user opens Home
**When** the client requests `/api/study-plan/daily?budget=20`
**Then** the hook returns loading, ready, empty, and error states
**And** the returned plan items preserve title, reason, estimated minutes, action URL, skill ids, priority, and completion state
**And** Home can retry the request after a failed load
**And** existing dashboard loading remains backward compatible.

### Story 21.3: Replace Manual Home Plan With Adaptive Plan

As a learner,
I want Home to show the most useful study tasks first,
So that I do not have to choose between many modules manually.

**Acceptance Criteria:**

**Given** the daily plan API returns one or more plan items
**When** Home renders the "Today's Plan" section
**Then** it displays adaptive plan items before manual fallback items
**And** each item shows reason text and estimated minutes
**And** each item links to its action URL
**And** if the daily plan API is empty or unavailable, Home falls back to current dashboard-based tasks
**And** existing streak, XP, vocabulary, leaderboard, and activity widgets remain visible.

### Story 21.4: Add Daily Session Start CTA

As a learner,
I want one primary button to begin today's study session,
So that I can start without scanning every module.

**Acceptance Criteria:**

**Given** Home has at least one uncompleted plan item
**When** the learner clicks the primary "Start Today's Session" action
**Then** the app opens the highest priority uncompleted plan item
**And** completed plan items are skipped
**And** if all items are complete, the CTA changes to a review or celebration state
**And** the CTA remains accessible on mobile and desktop.

### Story 21.5: Track Daily Plan Completion State

As a learner,
I want Home to reflect what I already completed today,
So that the plan feels responsive and current.

**Acceptance Criteria:**

**Given** a learner completes a plan-linked activity
**When** Home reloads or the plan is refetched
**Then** the relevant plan item is marked completed when the underlying activity log or learning event supports it
**And** completed state does not require manual user checking
**And** completion mapping is tested for at least vocabulary review, daily challenge, grammar quiz, and listening practice
**And** incomplete or unknown completion data fails gracefully.

## Epic 22: Unified Review Experience

Learners use one review entry point for due memory, mistakes, pronunciation, writing, listening, and reading tasks.

### Story 22.1: Add Due Review Task API Adapter

As a learner,
I want due review tasks to come from one queue,
So that review priority is consistent across modules.

**Acceptance Criteria:**

**Given** pending review tasks exist for a user
**When** the app requests due review data
**Then** the response includes task id, source type, source id, skill ids, priority, due time, estimated minutes, review mode, and reason metadata
**And** the API can still include legacy vocabulary and error review counts during migration
**And** unauthenticated requests return 401
**And** route tests cover empty, due, and unauthorized cases.

### Story 22.2: Build Today's Review Hub

As a learner,
I want one "Today's Review" page,
So that I can review what matters without choosing separate review modules.

**Acceptance Criteria:**

**Given** due review tasks exist
**When** the learner opens the review hub
**Then** tasks are grouped by learner need such as words to remember, mistakes to fix, pronunciation to drill, and passages to retry
**And** each group shows count, estimated time, and priority
**And** the learner can start a mixed review session
**And** legacy links to flashcards, review quiz, and error notebook still work.

### Story 22.3: Implement Mixed Review Session Shell

As a learner,
I want review tasks from different modules to appear in one session,
So that review feels like a coherent daily habit.

**Acceptance Criteria:**

**Given** due tasks include multiple source types
**When** the learner starts a mixed review session
**Then** the session presents one task at a time with a stable task header, answer area, and outcome controls
**And** vocabulary tasks can delegate to existing vocabulary/flashcard review behavior
**And** error tasks can delegate to existing error quiz behavior
**And** unsupported task types show a safe fallback link to the source module
**And** session progress and exit behavior are tested.

### Story 22.4: Close Review Outcome Loop From Unified Session

As a learner,
I want review outcomes to update future scheduling,
So that the app stops showing tasks I have already mastered.

**Acceptance Criteria:**

**Given** a learner completes a unified review task
**When** they mark the outcome as again, hard, good, or easy
**Then** the app updates the review task status or next due time
**And** a `review_completed` learning event is created when possible
**And** affected skill state is updated when possible
**And** failures in telemetry do not erase the learner's answer
**And** unit tests cover scheduling and mastery update outputs.

### Story 22.5: Add Review Entry Points To Home And Sidebar

As a learner,
I want one obvious place to review today's due tasks,
So that I do not miss important review work.

**Acceptance Criteria:**

**Given** review tasks are due
**When** Home and the sidebar render
**Then** the app shows a due review badge or plan item linked to the review hub
**And** the old flashcard and error badges remain available during migration
**And** the badge count does not double-count the same source item
**And** mobile and desktop navigation remain usable.

## Epic 23: Personal Error Remediation

Learners can see repeated mistakes, practice targeted drills, and measure error reduction.

### Story 23.1: Normalize Error Categories

As a learner,
I want my mistakes grouped into understandable categories,
So that I know which patterns to fix first.

**Acceptance Criteria:**

**Given** errors are created from grammar, writing, listening, daily challenge, mock test, or speaking feedback
**When** the error is stored or displayed
**Then** it includes or derives a normalized category such as tense, article, preposition, word form, coherence, pronunciation, vocabulary, listening detail, or exam strategy
**And** existing error records still display without migration
**And** new categories map to skill ids when possible
**And** classification fallback behavior is tested.

### Story 23.2: Add Error Pattern Summary

As a learner,
I want to see my most repeated error patterns,
So that I can focus on the highest-impact fixes.

**Acceptance Criteria:**

**Given** the learner has unresolved or repeated errors
**When** Error Notebook loads
**Then** it shows top error patterns by frequency and recency
**And** each pattern includes examples, affected skill, and recommended next action
**And** resolved errors are not hidden from trend calculations unless explicitly filtered
**And** empty state guidance appears for learners with no errors.

### Story 23.3: Generate Targeted Error Drills

As a learner,
I want drills generated from my actual mistakes,
So that practice directly fixes my weak patterns.

**Acceptance Criteria:**

**Given** an error pattern has enough source examples
**When** the learner starts a targeted drill
**Then** the app generates or selects practice items that test the same pattern
**And** every drill item includes answer validation and an explanation
**And** drill completion creates review tasks or learning events where appropriate
**And** AI-generated drills are schema-validated before display.

### Story 23.4: Show Error Improvement Trend

As a learner,
I want to know whether repeated errors are decreasing,
So that I can see real improvement beyond XP.

**Acceptance Criteria:**

**Given** the learner has historical error data
**When** Progress or Error Notebook renders improvement insights
**Then** it shows repeated-error trend by category over time
**And** it highlights categories that improved, worsened, or need review
**And** the trend uses available data without requiring destructive backfill
**And** the UI explains low-confidence data when there are few attempts.

### Story 23.5: Connect Error Notebook To Daily Plan

As a learner,
I want repeated errors to influence my daily plan,
So that the app helps me fix what I keep missing.

**Acceptance Criteria:**

**Given** the learner has high-priority repeated errors
**When** the daily plan is generated
**Then** error remediation candidates can appear as high or medium priority tasks
**And** the reason text names the pattern in learner-friendly language
**And** completing an error drill can reduce or reschedule the related review task
**And** recommendation tests cover repeated-error priority.

## Epic 24: Onboarding, Goals, and Adaptive Pathways

Learners choose a goal, set constraints, and receive a focused starter pathway.

### Story 24.1: Build Goal And Time Budget Onboarding

As a new learner,
I want to tell the app my goal and daily time,
So that recommendations match why I am learning English.

**Acceptance Criteria:**

**Given** a learner has no onboarding baseline
**When** they finish sign-in or open Home
**Then** the app offers a short onboarding flow for primary goal, daily time budget, weak skill, and preferred learning style
**And** the learner can skip and use defaults
**And** saved answers persist to onboarding baseline or preferences
**And** the flow is short enough to complete in under two minutes.

### Story 24.2: Connect Diagnostic To Baseline Skill State

As a learner,
I want my diagnostic result to initialize my learning profile,
So that my first recommendations are not generic.

**Acceptance Criteria:**

**Given** the learner completes diagnostic questions
**When** the diagnostic result is saved
**Then** baseline scores are mapped to skill ids
**And** user skill states are created or updated using the baseline confidence
**And** diagnostic failure does not block normal app usage
**And** tests cover baseline creation for grammar, vocabulary, listening, and reading.

### Story 24.3: Generate 7-Day Starter Pathway

As a learner,
I want a short first-week plan,
So that I know how to begin without choosing every module myself.

**Acceptance Criteria:**

**Given** the learner has a goal and baseline
**When** the starter pathway is generated
**Then** it contains seven days of suggested actions with estimated minutes and skill focus
**And** each day can be converted into daily plan candidates
**And** pathways exist for TOEIC, IELTS, daily conversation, workplace English, and foundation rebuilding
**And** the pathway adjusts when daily time budget is 5, 10, 15, 20, or 30 minutes.

### Story 24.4: Add Pathway View To Home

As a learner,
I want to see where today's tasks fit in my weekly pathway,
So that I understand the bigger plan.

**Acceptance Criteria:**

**Given** a starter pathway exists
**When** Home renders the plan area
**Then** it shows current day, goal, completed days, and next milestone
**And** it does not bury the primary daily action
**And** learners can change goal or time budget from a clear settings entry point
**And** the pathway view has an empty state for skipped onboarding.

### Story 24.5: Replan After Goal Or Time Budget Changes

As a learner,
I want the app to adapt when my goal or available time changes,
So that the plan remains realistic.

**Acceptance Criteria:**

**Given** the learner edits goal or daily time budget
**When** the changes are saved
**Then** future plan generation uses the new settings
**And** completed historical activities remain intact
**And** current-day tasks are refreshed safely
**And** tests cover switching between TOEIC and IELTS goals.

## Epic 25: Cross-Module Feedback and AI Coach

Learning sessions end with consistent feedback, next actions, and optional AI coach support.

### Story 25.1: Define End-Of-Lesson Summary Contract

As a learner,
I want each completed activity to summarize what happened and what to do next,
So that feedback leads to action instead of stopping at a score.

**Acceptance Criteria:**

**Given** a module completes a learning session
**When** it produces a summary
**Then** the summary includes result, explanation, top issue, next action, skill ids, and optional review task candidates
**And** the contract supports grammar, listening, reading, writing, speaking, pronunciation, vocabulary, and mock test modules
**And** invalid summary payloads fail validation
**And** no existing module UI is required to migrate in this story.

### Story 25.2: Apply Summary Pattern To High-Value Modules

As a learner,
I want feedback to look and behave consistently across modules,
So that I can quickly understand results and continue studying.

**Acceptance Criteria:**

**Given** the learner completes grammar quiz, listening exercise, writing review, pronunciation practice, or reading cloze
**When** the result screen appears
**Then** it follows the result, explanation, next-action pattern
**And** the next action can link to review, retry, dictionary, chatbot, or daily plan
**And** existing detailed feedback remains available
**And** tests cover at least three migrated module summaries.

### Story 25.3: Persist Versioned AI Feedback Runs

As a learning system,
I want AI feedback metadata stored consistently,
So that feedback quality and downstream review tasks can be audited.

**Acceptance Criteria:**

**Given** an AI scoring or feedback route produces structured output
**When** the output is accepted
**Then** the route stores template version, rubric version, model name, prompt hash, input snapshot, structured output, latency, cost estimate if available, and safety flags
**And** malformed AI output is rejected or transformed through a safe fallback
**And** failures to persist metadata do not lose the learner's visible feedback
**And** tests cover at least writing and speaking feedback routes.

### Story 25.4: Add Coach Summary To Chatbot

As a learner,
I want the chatbot to understand my recent study context,
So that it can coach me instead of only answering isolated questions.

**Acceptance Criteria:**

**Given** the learner has recent plan, review, error, or progress data
**When** they start a coaching conversation
**Then** the chatbot can summarize recent strengths, weak spots, and recommended next actions
**And** the prompt stays concise and avoids exposing raw private database details
**And** coach recommendations link to real modules
**And** normal chat behavior remains available.

### Story 25.5: Capture Chat Corrections As Learning Signals

As a learner,
I want corrections from chatbot practice to feed my learning profile,
So that informal practice improves future recommendations.

**Acceptance Criteria:**

**Given** the chatbot corrects grammar, vocabulary, pronunciation, or writing issues
**When** a correction is structured enough to act on
**Then** the app can save it as a learning event, error log entry, or review task candidate
**And** the learner is not interrupted by telemetry work
**And** low-confidence corrections are not forced into the review queue
**And** tests cover at least one saved grammar correction and one saved vocabulary correction.

## Epic 26: Progress Insights and Motivation

Learners see evidence of English improvement and understand the next best action.

### Story 26.1: Build Skill Mastery Progress Panel

As a learner,
I want to see my current skill strengths and weak spots,
So that I can understand why the app recommends certain tasks.

**Acceptance Criteria:**

**Given** user skill state exists
**When** Progress loads
**Then** it shows skill mastery for vocabulary, grammar, listening, reading, writing, speaking, and pronunciation
**And** each skill includes proficiency, confidence, last practiced time, and next recommended review if available
**And** the panel handles missing or low-confidence data gracefully
**And** the same data can inform Home recommendation explanations.

### Story 26.2: Add Learner-Friendly Improvement Statements

As a learner,
I want progress described in practical language,
So that I can feel real improvement beyond points.

**Acceptance Criteria:**

**Given** progress data is available
**When** the app renders progress insights
**Then** it can show statements such as remembered words, reduced repeated errors, reading level growth, longer speaking duration, or better listening speed
**And** every statement links to the underlying evidence category
**And** the app avoids overstating improvement when data is sparse
**And** copy is understandable to Vietnamese learners.

### Story 26.3: Add Review Debt And Memory Risk Insight

As a learner,
I want to know what I am likely to forget soon,
So that I prioritize review before learning new content.

**Acceptance Criteria:**

**Given** due or overdue review tasks exist
**When** Home or Progress renders review insight
**Then** the app shows review debt by category and estimated minutes
**And** high memory-risk items can appear in the daily plan
**And** the insight links to the review hub
**And** tests cover overdue review priority.

### Story 26.4: Add Weekly Learning Retrospective

As a learner,
I want a weekly summary of what improved and what to do next,
So that I can stay motivated and adjust my study habit.

**Acceptance Criteria:**

**Given** the learner has at least one week of activity
**When** the weekly retrospective is generated
**Then** it summarizes completed actions, strongest skill, weakest skill, repeated error pattern, review debt, and next week recommendation
**And** it can be displayed on Progress or sent as an optional notification/email in the future
**And** it does not require an AI call for the deterministic baseline version
**And** tests cover summary generation with sparse and rich data.

### Story 26.5: Rebalance Motivation Widgets Around Learning Outcomes

As a learner,
I want streaks and XP to support learning rather than distract from it,
So that I focus on consistent improvement.

**Acceptance Criteria:**

**Given** Home and Progress include streak, XP, badges, and leaderboard
**When** outcome insights are available
**Then** the UI gives priority to today's action, skill progress, and review needs before secondary gamification widgets
**And** existing gamification widgets remain accessible
**And** new empty states explain how to generate useful progress data
**And** mobile layout keeps the primary action visible without excessive scrolling.

## Validation Summary

- Every functional requirement is covered by at least one story.
- Each epic delivers user-visible value independently.
- Stories build on existing Epic 20 infrastructure without requiring destructive migrations.
- Known near-term risks are captured as stories: daily plan integration, recommendation route audit, unified review entry point, and cross-module feedback contracts.
- Existing app behavior is preserved during incremental migration.
